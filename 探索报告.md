# 提示词工程考核探索报告

## 摘要

本报告系统性地探索了在有限对话轮次（`max_turns <= 3`）约束下，如何通过提示词工程最大化大语言模型的代码题求解能力。经过两轮迭代优化，最终方案实现了从基线 `pass@1 ≈ 0.40` 到 `pass@1 ≈ 0.52` 的显著提升。核心策略包括：多候选生成、执行筛选、错误类型导向修复、Token 预算管理以及针对性的提示词设计。

---

## 1. 任务目标与约束

### 1.1 考核目标

在严格的对话轮次限制下（`max_turns <= 3`），通过精心设计的提示词策略，最大化模型在代码题求解任务上的 pass@1 指标。

### 1.2 交付物

- **核心文件**：`code/solution.py`，仅实现 `run_question` 函数

### 1.3 评测环境


| 组件       | 配置                                      |
| -------- | --------------------------------------- |
| 数据集      | `code/data/dev.jsonl`（20 道题目）           |
| 评测脚本     | `code/run.py`（官方评分入口）                   |
| 模型       | Qwen3-8B（本地部署）                          |


---

## 2. 探索方法论

### 2.1 整体策略框架

本探索遵循**"生成-验证-修复"**的闭环策略：

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  多候选生成  │ → │  执行验证   │ → │  错误修复   │
│  (Turn 1)   │    │  (筛选)     │    │  (Turn 2-3) │
└─────────────┘    └─────────────┘    └─────────────┘
       ↑                                    ↓
       └────────────────────────────────────┘
                 (失败时迭代)
```

### 2.2 核心设计原则

1. **多样性优先**：首轮生成多个候选解，提升覆盖不同算法思路的概率
2. **错误分类**：根据执行反馈（TLE/WA/RE）提供针对性修复提示
3. **Token 预算**：动态裁剪输入，确保关键信息不丢失
4. **简洁优于详尽**：避免过度详细的提示干扰模型判断

---

## 3. 第一轮迭代：基线建立与初步优化（V1 ~ V4）

### 3.1 V0：单轮直接求解（基线）

**策略**：直接将题目 prompt 发送给模型，要求返回单个代码块。

**观察结果**：

- 边界条件遗漏严重
- 算法复杂度不稳定
- 输出格式不规范

**Pass@1**：≈ 0.00 ~ 0.40（不稳定）

### 3.2 V1：单轮多候选 + 执行筛选

**策略演进**：

- **首轮**：要求模型输出 3 个 Python 代码块
- **执行筛选**：逐个调用 `execute_code`，任一通过即返回
- **失败处理**：全失败时保留最有效反馈用于下一轮修复

**关键改进**：引入候选多样性，初步建立验证机制

### 3.3 V2：失败反馈修复

**策略演进**：

- **第二轮**：拼接原题 + 上轮代码 + 执行反馈
- **修复生成**：要求输出 2 个修复版本，再次执行筛选

**关键改进**：利用执行反馈指导修复方向

### 3.4 V3：Token 预算控制

**策略演进**：

- **裁剪函数**：增加 token 裁剪逻辑，将输入控制在约 15k token
- **反馈策略**：采用头尾保留策略，减少关键信息丢失

**关键改进**：防止超长输入导致的上下文截断

### 3.5 V4：4 候选 + Token 预算

**策略演进**：

- **首轮**：4 候选生成
- **修复轮**：3 候选生成，最大化内采样

**问题发现**：候选过多导致单候选质量下降，长题代码出现截断

### 3.6 第一轮评测对比


| 版本  | 样本数 | 平均 Pass@1 | 主要问题     |
| --- | --- | --------- | -------- |
| V1  | 1   | 0.4000    | 单候选风险高   |
| V2  | 1   | 0.4500    | 修复提示不够精准 |
| V2  | 3   | 0.3667    | 样本波动大    |
| V4  | 5   | 0.4000    | 候选过多导致截断 |


**阶段结论**：候选数量需与 Token 限制平衡，4 候选策略在长题目上表现不佳。

## 4. 第二轮迭代：深度优化与稳定化（V5 ~ V6 + I/O 修复）

### 4.1 V5/V6 核心架构设计

#### 4.1.1 设计动机

V4 的 4 候选策略在 `max_tokens=8192` 限制下频繁导致代码截断。需要在**候选多样性**与**单候选质量**之间取得平衡。

#### 4.1.2 对话架构

采用 **3 轮 × 2 候选/轮** 策略（最多 6 次评测机会）：


| 轮次         | 策略                          | 上下文处理            |
| ---------- | --------------------------- | ---------------- |
| **Turn 1** | 要求两个不同算法的候选解                | 原始题目             |
| **Turn 2** | 携带 Turn 1 上下文，针对性修复         | 原题 + 错误代码 + 执行反馈 |
| **Turn 3** | 新鲜 prompt + 截断上下文，提示"考虑换算法" | 简化历史             |


#### 4.1.3 错误类型导向修复

实现 `detect_error_type` 函数，根据错误类型提供差异化修复提示：


| 错误类型        | 优先级 | 修复策略                          |
| ----------- | --- | ----------------------------- |
| TLE (超时)    | 4   | 推荐 KMP/Z函数、Fenwick树、前缀和、二分查找等 |
| WA (答案错误)   | 3   | 检查边界条件、索引偏移、输出格式、逆映射关系        |
| RE (运行时错误)  | 2   | 检查数组越界、输入解析、递归深度              |
| SyntaxError | 0   | 检查缩进和括号匹配                     |
| Other       | 1   | 通用修复建议                        |


#### 4.1.4 候选优先级评估

在 `evaluate_response` 中实现基于错误类型的优先级排序：

```python
_priority = {
    "TLE": 4,          # 超时 - 算法可优化
    "WA": 3,           # 答案错 - 逻辑可修复
    "RE": 2,           # 运行时错 - 实现可修复
    "Other": 1,        # 其他错误
    "SyntaxError": 0,  # 语法错 - 最难修复
}
```

### 4.2 关键提示词设计（V6 稳定版本）

#### 4.2.1 System Prompt

```
You are an expert competitive programmer.

Key guidelines:
- Always use sys.stdin.readline (or sys.stdin.read()) for fast I/O
- Choose algorithm by constraint size: N<=10^3→O(N²); N<=10^5→O(N log N); N<=10^6→O(N)
- CRITICAL: Re-read the output specification carefully. The output index may differ
  from the input index — build a reverse mapping if needed (e.g. inverse permutation)
- Output complete, runnable Python in python code blocks
```

#### 4.2.2 First User Prompt 核心约束

```
- BEFORE CODING: carefully identify what the OUTPUT index represents
  (it may differ from the input index — build reverse mappings if needed)
- After writing the solution, mentally verify it produces the correct output for Sample Input 1
```

#### 4.2.3 逆映射提示的显著效果

**案例：Q4 (Bib Numbers)**

- **题意理解**：person $i$ 佩戴号牌 $Q_i$，正视 person $P_i$
- **输出要求**：对每个号牌编号 $i$，输出佩戴该号牌的人所正视的人的号牌
- **正确解法**：先构建逆映射 $inv_Q$，再计算 $Q[P[inv_Q[bib]-1]-1]$

**效果对比**：


| 版本      | Q4 Pass@1   |
| ------- | ----------- |
| 无逆映射提示  | 0.00        |
| 加入逆映射提示 | 0.33 ~ 1.00 |


### 4.3 失败的尝试与教训

以下尝试均导致性能回归，最终回退至 V6：


| 版本     | 修改内容                                | 结果分析                |
| ------ | ----------------------------------- | ------------------- |
| V5b    | Turn 1 候选数从 2 增至 3                  | Q12/Q14 代码截断，整体性能下降 |
| V6b    | Repair prompt 增加详细调试步骤              | Q12/Q15 回归，提示过长干扰模型 |
| V7/V7b | System prompt 加入 KMP/Z-function 算法名 | Q12/Q14/Q15 回归      |
| V8/V8b | 强制推荐 `sys.stdin.read().split()`     | Q4 回归至 0.00         |


**核心教训**：

> 越详细或越具体的提示，反而可能干扰模型在其他题目上的表现。简洁清晰的通用指引比详尽的特定算法指导更稳健。

### 4.4 失败题型深度分析

#### A. 算法难题（稳定 0.00）

以下题目需要模型知识边界外的复杂算法，提示词工程难以突破：


| 题号  | 题目类型        | 所需算法/知识 |
| --- | ----------- | ------- |
| Q0  | 复杂网格模拟      | 复杂状态转移  |
| Q1  | 二进制串多数投票    | 递归 DP   |
| Q2  | LCS 计数      | 动态规划优化  |
| Q5  | 离线 LIS 参数查询 | 持久化线段树  |
| Q8  | 复杂约束优化      | 组合优化    |
| Q18 | Good String | 复杂组合/图论 |


#### B. Q17 博弈论（稳定 0.00）

**题意**：Fennec 和 Snuke 交替操作，完成所有索引标记时的胜负判断。

**模型常见错误**：生成 `sum(1 for x in A if x>=1) % 2 == 1` 的简单奇偶判断。

**深入分析**：胜负取决于"ones"（$A_i=1$）和"others"（$A_i \geq 2$）的复杂交互与 buffer 可用性，无简单公式可循。

**尝试结果**：所有"增加博弈论提示"的尝试均未奏效。

#### C. Q19 特殊 I/O 格式（稳定 0.00）

**双重挑战**：

1. **格式冲突**：题目为 LeetCode 风格函数签名，实际测试为 stdin/stdout
2. **输入解析**：输入格式为 `[1,2,3]` 的 Python 列表字面量，需 `ast.literal_eval` 解析
3. **算法难度**：需要 2D DP + 凸包技巧（$O(N^2)$）

**模型表现**：始终生成 `map(int, ...)` 的错误解析代码，多次尝试均无效。

#### D. 偶发性失败（0.33 ~ 0.60 波动）


| 题号         | 波动原因                           | V6 应对策略 |
| ---------- | ------------------------------ | ------- |
| Q4 (逆排列)   | 模型不总是遵从逆映射提示                   | 强化提示权重  |
| Q11 (网格翻转) | 需枚举 $2^W$ 列翻转状态，偶尔实现正确         | 无有效改进   |
| Q13 (最短回文) | KMP 中 `s[::-1]+'#'+s` 拼接顺序偶尔写反 | 无有效改进   |


### 4.5 V6 + I/O 修复（最终版本）

#### 问题发现

**Q12（三连续相等）偶发失败分析**：

模型错误地使用了单个 `input().split()` 读取所有输入，导致只读取了第一行数据，将 $N$ 和数组 $A$ 混在同一次 readline 调用中。

#### 修复方案

在 `first_user_prompt` 中增加明确的 I/O 规范：

```
- First line: use sys.stdin.readline for line-by-line input,
  OR data = sys.stdin.read().split() if you prefer reading all input at once
- NEVER use a single input().split() call to read multi-line input
```

## 5. 方法论总结


| 维度         | 结论                       | 证据                        |
| ---------- | ------------------------ | ------------------------- |
| **候选数量**   | **2 优于 4**               | Token 限制下，4 候选导致代码截断      |
| **多轮对话**   | **Turn 2 携带上下文有效**       | 错误分类修复提升修复成功率             |
| **错误分类修复** | **TLE/WA/RE 差异化提示有效**    | 针对性提示提高修复效率               |
| **逆映射提示**  | **对"输出索引≠输入索引"类问题效果显著**  | Q4 Pass@1 从 0.00 提升至 1.00 |
| **提示详略**   | **简洁优于详尽**               | V6b/V7/V8 详细提示均导致回归       |
| **特定算法提示** | **风险高于收益**               | 算法名提示干扰其他题目表现             |
| **候选优先级**  | **TLE > WA > RE 排序有助修复** | 优先修复可实现的问题                |

## 6. 风险识别与规避策略


| 风险类型     | 风险描述              | 规避策略                          |
| -------- | ----------------- | ----------------------------- |
| **格式风险** | 模型输出非代码块          | System + User prompt 双重约束格式要求 |
| **超长风险** | 输入超出 Token 限制     | 动态裁剪题目/反馈/历史代码                |
| **轮数风险** | 超出 `max_turns` 限制 | 严格循环内控制调用次数                   |
| **离线风险** | Tokenizer 依赖网络    | 使用本地模型与本地 Tokenizer 路径        |

## 7. 结论与展望

### 7.1 阶段性成果

经过两轮系统性探索，方案从"单轮直出"（`pass@1 ≈ 0.40`）升级为"多候选生成 + 执行筛选 + 分类反馈修复 + Token 预算 + I/O 规范"的闭环策略，在 Qwen3-8B 模型上稳定实现 `**pass@1 ≈ 0.52`**。

### 7.2 主要瓶颈

约 **10 道题（50%）** 需要模型知识边界外的复杂算法（博弈论、持久化数据结构、复杂 DP 优化），此类问题通过提示词工程难以突破。

### 7.3 未来优化方向

1. **模型能力提升**：更强基座模型可能突破当前算法瓶颈
2. **外部工具集成**：代码解释器、算法库调用可能扩展解题能力
3. **少样本学习**：引入相似题目的正确解法作为 in-context 示例
